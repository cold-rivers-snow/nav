# 需求文档

## 介绍

本规范定义了基于 WebStack-Hugo 主题创建自定义标签导航网站的需求，包括 GitHub Pages 部署和自定义内容，以替换现有内容为个性化的导航链接。

## 术语表

- **导航网站**: 基于 Hugo 的静态网站，显示分类链接
- **GitHub_Pages**: GitHub 的静态网站托管服务
- **Hugo**: 用于构建网站的静态网站生成器
- **WebStack_主题**: 提供导航网站布局的 Hugo 主题
- **标签分类**: 有组织的导航链接组（如开发工具、设计资源）
- **导航链接**: 包含图标、描述和 URL 的单个网站链接
- **配置文件**: 定义网站结构和内容的 Hugo 配置文件和数据文件
- **书签文件**: 浏览器导出的 HTML 格式书签文件
- **书签解析器**: 解析浏览器书签文件并提取链接信息的工具
- **去重系统**: 检测和移除重复 URL 的功能模块
- **临时书签页**: 显示当前工作相关链接的临时存储区域

## 需求

### 需求 1: 网站内容自定义

**用户故事:** 作为网站所有者，我希望用自己的个性化导航链接替换现有内容，以便网站反映我偏好的工具和资源。

#### 验收标准

1. 当访问网站时，导航网站应显示自定义分类而不是当前的分类
2. 当用户查看任何分类时，导航网站应显示带有相关描述的个性化链接
3. 当显示图标时，导航网站应为每个服务使用适当的图标或提供备用图像
4. 导航网站应保持现有的视觉设计和布局结构
5. 当显示网站标题时，导航网站应显示自定义标题

### 需求 2: GitHub Pages 部署

**用户故事:** 作为网站所有者，我希望将网站部署到 GitHub Pages，以便公开访问并在我进行更改时自动更新。

#### 验收标准

1. 当代码推送到主分支时，GitHub_Pages 应自动构建和部署网站
2. 当部署完成时，导航网站应通过 GitHub Pages URL 可访问
3. 当 Hugo 构建网站时，配置文件应在正确的输出目录中生成静态文件
4. GitHub_Pages 应使用适当的 HTTPS 提供网站服务，如果配置了自定义域名则支持自定义域名
5. 当构建错误发生时，GitHub_Pages 应提供清晰的错误消息并优雅地失败

### 需求 3: 导航分类组织

**用户故事:** 作为用户，我希望浏览按相关分类组织的导航链接，以便快速找到所需的工具和资源。

#### 验收标准

1. 当网站加载时，导航网站应显示至少 5 个主要链接分类
2. 当展开分类时，导航网站应显示子分类（如果适用）
3. 当显示链接时，导航网站应为每个链接显示标题、描述和图标
4. 导航网站应包括以下分类：开发工具、设计资源、学习资源、生产力工具和娱乐
5. 当用户点击导航链接时，导航网站应在新标签页中打开目标 URL

### 需求 4: 网站配置管理

**用户故事:** 作为网站维护者，我希望通过 YAML 文件轻松更新网站配置和内容，以便在不接触 HTML 或复杂代码的情况下修改网站。

#### 验收标准

1. 当对 config.toml 进行配置更改时，导航网站应反映更新的网站设置
2. 当对 webstack.yml 进行内容更改时，导航网站应显示更新的导航链接
3. 当新图标添加到 assets 目录时，导航网站应正确显示它们
4. 配置文件应支持轻松添加新分类和链接
5. 当提供无效配置时，Hugo 应在构建期间提供清晰的错误消息

### 需求 5: 响应式设计和性能

**用户故事:** 作为用户，我希望导航网站在所有设备上都能正常工作并快速加载，以便我可以从任何地方访问我的书签。

#### 验收标准

1. 当在移动设备上访问网站时，导航网站应显示格式正确的内容
2. 当网站加载时，导航网站应在 3 秒内完成初始渲染
3. 当加载图像时，导航网站应使用懒加载以获得更好的性能
4. 导航网站应保持现有的响应式设计功能
5. 当访问网站时，导航网站应在所有现代浏览器中正常工作

### 需求 6: 搜索和导航功能

**用户故事:** 作为用户，我希望搜索导航链接并轻松浏览分类，以便快速找到特定工具或发现新资源。

#### 验收标准

1. 当用户在搜索框中输入时，导航网站应根据标题和描述过滤链接
2. 当显示搜索结果时，导航网站应突出显示匹配的术语
3. 当没有找到搜索结果时，导航网站应显示有用的消息
4. 导航网站应保持现有的侧边栏导航功能
5. 当点击分类时，导航网站应平滑地展开/折叠部分

### 需求 7: 临时工作书签页

**用户故事:** 作为用户，我希望有一个临时工作书签页来显示当前正在使用的网站链接，以便快速访问当前工作相关的资源，并可以随时删除或移动到收藏页中。

#### 验收标准

1. 当访问网站时，导航网站应显示一个"当前工作"或"临时书签"分类
2. 当添加临时书签时，系统应支持快速添加当前正在使用的网站链接
3. 当临时书签过多时，系统应提供批量管理功能（删除、移动到其他分类）
4. 当用户完成工作时，系统应支持一键清空临时书签页
5. 当临时书签存在时，系统应在页面上提供明显的视觉标识表明这是临时内容
6. 系统应支持将临时书签拖拽到其他分类中进行永久保存
7. 当临时书签页为空时，系统应显示提示信息指导用户如何添加临时书签

### 需求 8: 浏览器书签导入功能

**用户故事:** 作为用户，我希望能够导入浏览器导出的书签文件，并自动添加到导航网站中，同时去除重复的网址，以便快速迁移我的书签收藏。

#### 验收标准

1. 当上传浏览器书签文件时，系统应解析 HTML 格式的书签文件
2. 当解析书签时，系统应提取网站标题、URL 和文件夹结构信息
3. 当检测到重复 URL 时，系统应自动去重并保留最新的条目
4. 当导入完成时，系统应将新书签按文件夹结构自动分类到相应的导航分类中
5. 当书签没有图标时，系统应尝试自动获取网站 favicon 或使用默认图标
6. 系统应支持 Chrome、Firefox、Safari 等主流浏览器的书签导出格式
7. 当导入过程中出现错误时，系统应提供详细的错误报告和建议

### 需求 9: GitHub Actions 工作流

**用户故事:** 作为开发者，我希望通过 GitHub Actions 进行自动化部署，以便当我将更改推送到仓库时网站自动更新。

#### 验收标准

1. 当代码推送到主分支时，GitHub_Actions 应触发 Hugo 构建过程
2. 当构建成功完成时，GitHub_Actions 应部署到 GitHub Pages
3. 当构建失败时，GitHub_Actions 应提供详细的错误日志
4. GitHub_Actions 应使用最新稳定版本的 Hugo 进行构建
5. 当部署完成时，GitHub_Actions 应在 5 分钟内更新实时网站
